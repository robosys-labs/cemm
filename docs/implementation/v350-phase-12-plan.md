# CEMM v3.5 Phase 12 — Cross-domain transition vertical-slice plan

**Status:** planned; not implemented by Phase 11
**Prerequisite:** Phase 11 generic transition engine phase-verified
**Authority:** test/promoted domain packages only; no new kernel ontology authority

## 1. Purpose

Phase 12 proves that the transition architecture is genuinely reusable. It must not become a disguised request to add a particular event, organism, device, organization, or lexical item to the structural foundation.

The proof succeeds only when several unrelated packages traverse the same Phase 2–11 machinery without any package-specific Python semantic branch.

## 2. Package-selection requirements

Choose at least three independently reviewed packages spanning different transition structures. Across the set, cover:

- irreversible terminal state change;
- reversible activation/deactivation or restoration;
- ordered/scalar/resource change;
- externally caused localization or movement;
- state-dependent capability change;
- lexical polysemy resolved by type/context rather than word-specific branching.

Packages may be synthetic competence fixtures or separately promoted learned data. They must not be inserted into the Phase-6 structural foundation solely to make the demo pass.

## 3. Required path per package

Each package must exercise the longest applicable path:

```text
raw evidence
-> Phase-7 form/sense evidence
-> Phase-8 grounding
-> Phase-9 UOL composition
-> Phase-10 attributed claim/epistemic placement when applicable
-> exact admitted target-context event occurrence
-> Phase-11 transition-contract compilation
-> non-mutating transition preview
-> proof-bearing StateDelta candidates
-> immutable state timeline projection
-> capability dependency reevaluation
-> atomic CAS commit
-> restart
-> history/query verification
```

No test may bypass earlier phases by constructing only the expected final state unless that test is explicitly a lower-level unit test.

## 4. Contrast matrix

For every applicable event/package, include:

| Contrast | Required result |
|---|---|
| admitted positive occurrence | eligible for transition subject to contract/preconditions |
| truth-negated proposition | no transition |
| possible/modal occurrence | no actual-context transition |
| hypothetical/counterfactual | isolated; no actual-context transition |
| prevented/failed/non-occurring | no resulting effect |
| attributed report without target admission | no target-context transition |
| admitted attributed report | transition only in explicitly admitted target context |
| fictional/simulated context | effects remain inside that context |
| retracted admission before commit | commit rejected |
| changed pre-state after preview | stale plan/proof rejected |
| ambiguous grounding | preserve alternatives/frontier; no speculative transition |
| unresolved time | explicit frontier; no timeline mutation |
| restart | proof/history/current projections survive and re-query correctly |
| correction/retraction | dependent projections invalidate/recompute through generic lineage |

## 5. Genericity/adversarial tests

### 5.1 Kernel-diff test

Capture a hash or diff of semantic-authority kernel paths before adding each package. Adding/promoting a package must not require a semantic control-flow edit under:

```text
cemm/v350/grounding
cemm/v350/composition
cemm/v350/epistemics
cemm/v350/transitions
```

Changes to generic bug fixes are reviewed separately and cannot be justified solely by one package name.

### 5.2 Renaming test

Where practical, mechanically rename package semantic refs/keys/forms while preserving structure. Transition behavior must remain structurally equivalent. Failure indicates hidden name-based authority.

### 5.3 Cross-language equivalence

For semantic cases lexicalized in multiple reviewed language packages, equivalent selected UOL must produce equivalent transition previews/proofs independent of surface language.

### 5.4 Context isolation

Run identical semantic content in at least two contexts. A transition in one context must not mutate another absent an explicit independent admission/bridge.

### 5.5 Proof forgery

Attempt to alter:

- target state value;
- holder;
- state dimension;
- contract revision;
- admission revision;
- pre-state revision;
- effective time;

while retaining a real event/proof lineage. Commit validation must reject the mismatch.

## 6. Capability tests

For packages with state-dependent capabilities:

- transition contract produces only state effects;
- separate capability dependency record observes post-transition projected state;
- capability instance revision changes immutably;
- designed function does not disappear merely because capability becomes unavailable;
- a type that never had an affordance does not acquire/lose it because another package does.

## 7. Movement tests

Where movement/localization is covered:

- self-initiated action capability and externally caused occurrence remain distinct;
- inability to self-initiate does not prohibit externally caused movement;
- localization change requires a generic representable location/state/relation lifecycle path;
- if that generic representation is not yet available, emit a frontier instead of package-specific mutation code.

## 8. Relation/role lifecycle gate

Phase 11 intentionally does not fake relation/role effects as state strings. Before a Phase-12 package requires relation/role creation/termination, implement and separately verify first-class generic relation/role delta/lifecycle records, persistence, validation, proof lineage, and timeline projection.

Until then such a requirement must remain an explicit blocked frontier.

## 9. Impact boundary

Phase 12 may collect consequence evidence for later impact tests, but transition contracts must not encode stakeholder valence/importance as state effects. Impact remains a later contextual assessment.

## 10. Persistence/restart gate

After commit and restart verify:

- event occurrence and admission lineage;
- exact transition proof;
- historical pre-state assignment revision;
- active post-state assignment;
- state/capability deltas;
- capability instance revision;
- dependency fingerprints/invalidation behavior.

## 11. Performance evidence

Capture per-stage timing for:

- contract discovery/compile cache;
- admission validation;
- condition evaluation;
- preview/proof generation;
- timeline projection;
- capability dependency evaluation;
- CAS commit validation/write.

Performance limits are budgets, never permission to skip proof or context checks. Timeout must preserve a frontier/partial result.

## 12. Phase-12 exit gate

Phase 12 passes only when:

1. at least three structurally distinct packages pass the full applicable pipeline;
2. no package-specific semantic kernel branch/template/enum addition is required;
3. all negative/context/modal/non-occurring contrasts block unsafe mutation;
4. all committed effects are reconstructible from exact reviewed contract + admission + pre-state revisions;
5. capability effects are dependency-driven;
6. restart/history/correction behavior is correct;
7. multilingual equivalent UOL yields equivalent transition semantics where applicable;
8. any still-unrepresentable relation/role/causal effect remains a declared frontier rather than a shortcut;
9. full repository CI and architecture/debt ratchets pass.
