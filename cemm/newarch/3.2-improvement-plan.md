# CEMM 3.2 Improvement Plan

Status: implementation record — supersedes v3.1 contract; all types, compilers, and runtime wiring implemented  
Target: CEMM v3.2 operational meaning and state-transmutation spine  
Scope: core loop, UOL operational meaning, causal/effect routing, query/write contracts, response preconditions, and conformance tests  
Non-goal: better surface responses by adding more answer templates

> **Implementation note:** This plan was originally written as a proposal. As of the current codebase, all types (§4), all kernel compilers (§8), all contract builders (§8), and the runtime integration (§7, §9) are implemented. The feature data plane (breaks 1-5), evidence explanation gating (§6.5), and blocked plan gating (§6.6) are fixed. See the "Current Implementation Gaps" section below for the few remaining open items.

## 1. Executive Correction

CEMM 3.1 completed the response-formation phases, schema-kernel refactor, graph-patch learning boundary, and budget-aware response/query pieces. The remaining failures are not NLG failures. They are failures in the operational meaning spine before NLG.

The 3.2 correction is:

```text
surface evidence
-> UOL atoms/edges/candidate sets
-> operational meaning frames
-> state transmutation frames
-> causal/effect routing
-> obligation contract
-> query/write/reaction contracts
-> response situation
-> response formation
```

Do not implement:

```text
surface text
-> intent label
-> response route
-> NLG template
```

The main architectural gap is that the runtime still collapses distinct operational meanings into broad lanes such as `assertion -> store_patch` or `question -> answer_concept`. 3.2 introduces explicit operational contracts so that profile facts, concept definitions, response feedback, dismissal, user state reports, safety risks, and social acts cannot collapse into one path.

## 2. State Transmutation Is Necessary

State transmutation is not optional. It is already implied by the architecture:

- `StateDimensionRegistry` defines state families and dimensions.
- `ActionOperatorRegistry` defines state and relation deltas.
- `AffordanceRegistry` predicts contextual effects.
- `ConceptAtom.state` defines lifecycle transitions.
- `GraphPatch` is the only legal durable mutation input.

The missing piece is a first-class runtime contract that says what kind of state is changing, whose state is changing, whether the change is observed, inferred, desired, predicted, committed, reversible, ephemeral, session-level, or durable.

Without this, the runtime treats feedback, facts, identity claims, affect, session exits, and memory writes as generic assertions.

## 3. Terminology

### 3.1 State Occupancy

A current or reported state value on an entity, concept, action, source, or conversation object.

Examples:

```text
user.identity.name = Chibueze
self.operational.status = online
conversation.session.status = active
user.affective.mood = positive
concept:president.informational.known = candidate
```

### 3.2 State Delta

A proposed change in state caused by an action, process, evidence update, affordance prediction, or correction.

Examples:

```text
memory_write -> self.cognitive.knowledge increase
user_feedback_too_verbose -> self.response_style.detail decrease
session_exit_request -> conversation.session.status set closing
fresh_source_retrieved -> claim.informational.verified increase
```

### 3.3 State Transmutation

A resolved transition from prior state to next state with scope, provenance, authority, and persistence policy.

```text
prior_state + delta + authority + policy -> next_state_candidate
```

A transmutation is not automatically durable. It may be:

```text
ephemeral
session_state
patch_candidate
durable_candidate
quarantined
rejected
```

### 3.4 Operational Meaning Frame

A runtime view compiled from UOL graph structure. It is not a new UOL atom kind. It is the planner-facing authority unit.

### 3.5 Effect Frame

A causal/effect prediction derived from operational meaning frames, state transmutation candidates, affordance rules, and source/evidence policy.

### 3.6 Obligation Contract

The authoritative decision contract that says what the system must do this turn. It replaces broad instruction-kind routing as the source of query/write/reaction behavior.

## 5. Architectural Fit

3.2 must not add UOL atom kinds or edge types.

State transmutation is represented in the working graph using existing primitives:

```text
state atoms
has_property edges
causes edges
enables/prevents edges
evaluates edges
before/after temporal edges
source/evidence/permission atoms
```

`StateTransmutationFrame` is a runtime contract derived from those graph structures. It is not a graph primitive.

Durable state changes still flow through:

```text
GraphPatch -> validation/scoring -> PatchCommitter -> durable semantic memory
```

Session and response-style state changes flow through:

```text
ReactionContract / OutputStateUpdate / SessionStore
```

They do not become concept-lattice facts unless explicitly authorized by a writable meaning frame.

## 6. Current Failure Classes And Required 3.2 Fixes

### 6.1 Profile State Lost

Failure:

```text
my name is Chibueze
-> generic has_property relation
-> property_dimension lost
-> what's my name? fails
```

3.2 fix:

```text
OperationalMeaningFrame.frame_type = profile_assertion

state_family = identity
dimension = name
proposed_value = Chibueze
persistence_policy = graph_patch_candidate

```

The feature chain must be preserved:

```text
UOLEdge.features
-> PatchOperation.fields["features"]
-> DurableRelationRecord.features
-> RelationFrame.features
-> SlotFill.features
-> BoundSlot.features
-> RealizationUnit.features
```

Validated break points (5 breaks in current 3.1 code):

```text
Break 1: MeaningGraphBuilder._extract_graph_patches (line ~1726)
  PatchOperation.fields does NOT copy edge.features.
  Fields dict has relation_key, surfaces, concept_ids but no "features" key.

Break 2: DurableRelationRecord (durable_semantic_store.py line ~12)
  Has NO features field at all. No relation_scope, no dimension.

Break 3: DurableSemanticStore.add_relation (line ~106)
  Method signature has NO features parameter.
  Even if the record had the field, it couldn't be populated.

Break 4: DurableSemanticStore.apply_validated_patch (line ~495)
  Extracts fields from op.fields but never reads a "features" key.

Break 5: DurableSemanticStore._record_to_frame (line ~447)
  Constructs RelationFrame WITHOUT passing features.
  RelationFrame.features defaults to {}.
```

Note: RelationFrame and SlotFill dataclasses already have `features` fields.
The chain is structurally ready — the gap is purely in the data plane.

Additional: RelationFrameCompiler._compile_relation_atom (line ~216) does NOT
preserve features, while _compile_edge (line ~213) does. This inconsistency
means relation-atom-compiled frames lose features even for current-turn data.

Failure:

```text

-> no concept-definition contract

-> unrelated self/name/capability answer
```


The query engine must never execute a broad fallback scan when the target concept is missing.

Validated: SemanticQueryEngine._match_frames_by_surface (line ~449) does
token overlap matching — "what does president mean?" can match any frame
containing "president" in subject/object surfaces.

Additionally, semantic_kernel_runtime.py (line ~322) does a BROAD ALL-RELATION
durable scan on every turn where the store has any relations:

This feeds unfiltered frames into the query builder and must be eliminated
(see §10.1). The §9.6 refactor must be linked to this runtime scan removal.

For self-query obligations, if no frames match by IDs, the engine falls back
to ALL answerable frames with subject.entity_id == "self" (line ~177), which
can return completely unrelated self-knowledge.

### 6.6 Blocked Plan Realized Before Gate

Failure:

```text

```

3.2 fix:

```text

```

Blocked plans must never reach `RealizationExecutor`.

Validated: ResponseFormationEngine.form (line ~44) realizes the plan (line 46)
BEFORE checking blocked (line 55). The text is overridden afterward, but:
1. RealizationExecutor still runs (wasted computation)
2. The `realized` object is used for action proposal at line 48:

```text
   In semantic_kernel_runtime.py, patches are extracted (step 5), validated

   uses raw instruction kind priority, not refined obligation priority.
```

## 7. 3.2 Runtime Loop

The efficient 3.2 runtime loop is:

```text
0. restore context/session state
1. perceive signal through language adapter
2. build meaning groups, hypotheses, candidate atoms
3. build UOLGraph with source/time/evidence/permission atoms
4. resolve concepts and ports
5. compile state occupancy and state-delta candidates
6. compile OperationalMeaningFrames
7. arbitrate meaning frames without collapsing candidate sets too early
8. compile StateTransmutationFrames
9. run causal/effect router
10. compile SemanticProgram from selected meaning frames
11. compile ObligationContract
12. extract authorized graph patches only from writable frames/contracts
13. validate and commit allowed patches
14. build QueryContract if required
15. execute budget-aware semantic query through contract constraints
16. apply session-level reaction/style/state transmutations
17. build ResponseSituation from contracts, evidence, write outcome, reaction state, safety state
18. generate/gate/rank/select response candidate plans
19. realize selected unblocked plan only
20. authorize internal actions
21. extract learning patch candidates
22. persist session state and traces
```

This preserves the existing architecture order:

```text
Observe -> Contextualize -> Interpret -> Ground -> Retrieve -> Infer -> Decide -> Realize -> Update -> Learn
```

but makes state transmutation and obligation contracts explicit before query/write/response.

Pass indexes to compilers instead of repeatedly scanning graph lists.

### 10.3 Cheap-To-Expensive Cascade

```text

-> current graph indexes
-> current session state
-> durable indexed lookup
-> source/tool retrieval if freshness required

```



## 17. Final 3.2 Contract

CEMM 3.2 is not a response improvement. It is the operational meaning and state-transmutation spine between UOL graph construction and response formation.

The final contract is:

```text
UOL graph is the temporary semantic workbench.
Operational meaning frames are the decision units.
State transmutation frames are the state-change authority units.
Obligation contracts decide query/write/reaction/safety behavior.
Graph patches remain the only durable learning mutation path.
Response formation renders only after these contracts are valid.
```
